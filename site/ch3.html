Chapter 3

The AI-Native Overhaul

“We changed five variables in a simulation of 45 people. Here is what the model produced. Here are the reasons it might be wrong.”

Chapter 2 established the IST: effective GP capacity at 62%. Treeknorm violations at 38%. System-failure deaths at 4.2 per run. Three interlocking failure mechanisms — administrative burden, waiting lists, data fragmentation — each making the others worse.

This chapter describes the SOLL. Same 45 citizens. Same 3 GPs. Same hospital. Same diseases, same comorbidities, same Markov chains. Five parameter changes. We measure what happens.

IST:  Ceff = Ctotal × (1 − 0.30 − 0.08) × 1.00 = 0.62
SOLL: Ceff = Ctotal × (1 − 0.05 − 0.08) × 1.34 = 0.83

From 0.62 to 0.83—a 34% increase in effective GP capacity without hiring a single additional doctor. A practice with a 2,400-patient norm goes from 1,488 effective slots to 1,992. Multiply across 12,000 Dutch practices: roughly 6 million additional patient-equivalent slots nationally.

Five Interventions

Each maps to a specific IST failure from Chapter 2. None require new medical knowledge, new drugs, or new buildings. All require restructuring how information flows.

#InterventionIST → SOLLMechanism
1 Ambient AI Scribes Ladmin: 30% → <5% Real-time documentation during consultations. GP speaks, system writes, FHIR resources auto-generated.
2 AI-Driven Triage Wait: 12wk → <4wk Triage agent queries specialist Agent Cards for capacity. Routes by ICD-10 match, queue length, urgency.
3 Digital Twin Predictions Reactive → Proactive Risk scores from FHIR history + comorbidity multipliers trigger GP consultation before symptoms manifest.
4 A2A Scheduling Isolated → Coordinated Provider agents share real-time availability. Referral round-trip: weeks → hours.
5 Hybrid openEHR + FHIR Interop: 11% → 66% openEHR for storage, FHIR R4 for exchange. Specialist sees full patient history on referral.

The residual 5% admin accounts for tasks requiring human judgment: signing off complex referrals, reviewing AI summaries, handling exceptions. Zero percent is not credible. Five percent is ambitious but defensible based on early deployments (Nuance DAX, Amazon HealthScribe).

The Digital Twin is not a large language model making clinical decisions. It is a risk calculator reading structured FHIR data. When Hendrik Veenstra’s twin calculates 32% heart failure probability (I25 + E11, COPD→CVD RR=2.5), it triggers a proactive GP visit. Intervention at “mild” rather than “severe” — cheaper, simpler, more effective.

The IZA projects needing 301,000 additional healthcare workers by 2035. This model suggests technology could offset approximately 110,000 through efficiency gains — not replacement, but augmentation. The remaining 191,000 still need to be trained, recruited, and retained.

The Agentic Mesh

The SOLL simulation runs on a three-layer architecture: data (FHIR R4), communication (A2A Protocol), and cognition (the cognitive loop). This section is for technical readers.

A2A Protocol — Agent Cards and referral lifecycle

Every provider publishes an Agent Card — a machine-readable description of capabilities, availability, and queue state:

{
  "agentId": "gp-de-jong",
  "name": "Dr. de Jong",
  "role": "gp",
  "skills": ["triage", "referral", "chronic_care"],
  "waitTime": { "weeks": 2, "queueSize": 12 },
  "metadata": {
    "admin_load": "5%",
    "treeknorm_status": "compliant",
    "nza_consult_rate_eur": 12.43,
    "max_patients": 2400
  }
}

Message types: referral, status_query, response, alert, dialogue.

Referral lifecycle: submitted → working → input-required → completed. Each transition generates a FHIR Encounter resource. In IST, a referral sits in “submitted” for 12 weeks. In SOLL, the A2A layer escalates automatically when wait times approach Treeknorm.

FHIR R4 Memory Store — Every action as a resource

Every agent action is stored as a FHIR R4 resource: Patient, Condition (ICD-10), Observation (HP, wait times), Encounter (visits), MedicationRequest (treatments).

Custom observation codes:

CodePurpose
CAMMELOT-WAITCurrent wait time in weeks
CAMMELOT-HPHealth points (0–100)
CAMMELOT-GHOSTMortality event marker
CAMMELOT-ACTIONAgent decision audit log

The “Memory Stream” (Park et al.) is a chronological FHIR query filtered by agent ID. An agent’s memories are its FHIR resources. Reflection operates on these. Planning generates new ones.

Implementation note: FHIR_STORE is a capped array (2,000 resources, FIFO eviction). A2A_MESSAGES caps at 200. Resources follow R4 structure but skip full schema validation — a known simplification.

The Cognitive Loop

Every agent executes a three-phase loop per tick, adapted from Park et al.’s “Generative Agents” (Stanford, 2023):

PhaseISTSOLL
1. Memory Stream Fragmented records, missing history Full FHIR history, all providers
2. Reflection “I’ve spent 30% on admin. I must rush triage.” “5% on admin. I can assess comorbidities properly.”
3. Planning Imprecise referral, no specialist visibility Optimal routing via A2A, risk-scored priority

The loop is identical in both modes. What changes is the information available during reflection. Better data produces better plans. Better plans produce better outcomes. The architecture is the intervention.

Limitations

A simulation is not evidence. It is a structured argument with explicit assumptions.

Seven caveats
  • N=45 is not N=5,000. Stochastic variance is substantial. A single severe comorbidity case shifts mortality by 20%. We run 200 simulations and report means. Confidence intervals are wide.
  • Eai = 1.34 is theoretical. Derived from IZA projections and pilot data, not from an RCT. Could be 1.15 (disappointing) or 1.60 (transformative). The simulation shows what happens if it holds.
  • Adoption barriers are not modeled. Physician trust, regulatory approval, patient acceptance, workflow integration — where most health-IT projects fail. Not in the model.
  • Financial incentives are absent. If AI frees 25% of GP time, does that become more patients (volume) or better care (quality)? We assume quality. Reality will be messier.
  • 66% interoperability is a policy goal. The IZA allocated €2.8B. Whether that produces actual interoperability depends on political will, vendor cooperation, and the definition of “interoperable.”
  • The cognitive loop is simplified. Three deterministic steps, not an LLM reasoning about patient care. Real AI systems capable of clinical reflection are research prototypes, not deployed products.
  • The author works at Microsoft, which sells AI and cloud products to healthcare organisations. This is a personal project. The code is open source. But framing choices are inevitably shaped by professional context.

The Point

The Dutch healthcare system faces a structural crisis that demographics will intensify and incremental reform cannot solve. The IST model quantifies the crisis. The SOLL model quantifies one possible response.

Whether these numbers are achievable depends on factors the simulation does not model: regulatory frameworks, physician culture, patient trust, vendor interoperability, insurance incentives, political will.

But the exercise has value precisely because it replaces opinion with parameters. You can disagree with Ladmin = 5% (too aggressive). You can challenge Eai = 1.34 (too optimistic). You can argue the Digital Twin thresholds are wrong. Good. That is the point.

The purpose of this simulation is to have numbers to argue about, not opinions.

The live simulation lets you toggle between IST and SOLL in real time. Watch the same 45 citizens under both regimes. Count the ghosts. Check the queues. Then argue about the parameters.
