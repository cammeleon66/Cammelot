Chapter 1
System Architecture

Inside the Engine Room

Cammelot isn’t a “black box” neural network. It is a decentralized operating system built on the premise that healthcare doesn’t have a data problem—it has a coordination problem.

Traditional healthcare IT is a “Tower of Babel.” In 2024, only 11% of Dutch health data was interoperable between providers. The IZA wants that at 66% by 2030, spending €2.8 billion on transformation plans. Most of those plans build point-to-point integrations: system A talks to system B through a custom connector. Repeat for every pair.

To fix Cammelot’s healthcare, we didn’t build a better database; we built a better protocol. The architecture rests on three pillars:

1. The Agentic Mesh (A2A Protocol)

In Cammelot, there is no central server “telling people where to go.” Every GP, Specialist, and Hospital is an Autonomous Agent. They communicate via the Agent-to-Agent (A2A) protocol.

The Agent Card

Every provider publishes a machine-readable “business card.” When a patient needs care, the system doesn’t search a static directory; it queries the mesh in real-time.

GET /.well-known/agent-card.json200 OK
{
  "agentId": "gp-de-jong",
  "name": "Dr. de Jong",
  "specialty": "General Practice",
  "currentWait": "14 days",
  "treeknorm_compliant": true,
  "capacity": 0.85,
  "nza_consult_rate_eur": 12.43
}

That’s it. No API contract negotiation. No integration middleware. Just: “here’s who I am, here’s what I can do, here’s how busy I am right now.” Any other agent in the mesh can read this and decide whether to send a referral.

The Referral Handshake:

Step 1   GP reads Cardiology Agent Card
GP → Hospital: { type: "referral", patient: "Hendrik", condition: "I25" }
Step 2   Hospital checks capacity & Treeknorm
Hospital → GP: { status: "accepted", estimatedWait: "3 weeks" }
Step 3   Referral lifecycle: submitted → working → completed

If the cardiologist’s queue is full, the referral bounces with a machine-readable rejection and Dr. de Jong auto-routes to the next available specialist. No fax machines involved.

Compare this to real Dutch integration: adding a new specialist system takes 3–18 months. In Cammelot, it takes 5 minutes—publish an Agent Card, join the mesh, start receiving referrals.

2. FHIR R4: The Universal Language

Every life in Cammelot is a stream of HL7® FHIR® R4 resources. This isn’t just for show—it is the bedrock of the simulation’s validity. Because we use the R4 standard, we can measure exactly how “data silos” lead to clinical errors.

FHIR Observation — COPD SeverityPatient/Hendrik-Veenstra
{
  "resourceType": "Observation",
  "status": "final",
  "code": { "coding": [{ "system": "http://cammelot.sim", "code": "COPD-stage" }] },
  "valueString": "Moderate",
  "subject": { "reference": "Patient/Hendrik-Veenstra" },
  "effectiveDateTime": "2026-04-15"
}

Memory as Data: Every visit is an Encounter. Every diagnosis is a Condition. Every prescription is a MedicationRequest. The entire life history of a pixel citizen is a chronological FHIR query—the same “Memory Stream” from Park et al.’s Generative Agents paper, except in structured clinical data instead of plain text.

Memory Stream QueryqueryFHIR()
function queryFHIR(agentId, type) {
  return FHIR_STORE.filter(r =>
    r.subject?.reference === 'Patient/' + agentId &&
    (!type || r.resourceType === type)
  );
}

The Digital Twin risk calculator reads this same store. No fancy ML model—just structured data, queried in real-time, producing actionable risk scores. When your data is standardized and queryable, even simple logic produces useful outputs.

3. The Cognitive Loop

The agents don’t just react; they reflect. Using a deterministic version of the Generative Agents architecture (Park et al., Stanford 2023), each agent executes a three-phase loop every 4 simulation ticks:

1
Perception
Read the FHIR record. “Patient HP is declining. Two conditions are active. Wait time exceeds Treeknorm.”
2
Reflection
Assess system constraints. “I am at 110% capacity. Administrative load is 30%. I must triage by severity.”
3
Action
Execute a clinical intervention or a referral handshake via the A2A mesh.

If a GP agent reflects that administrative load exceeds 30%, they begin to make triage errors. This is how Cammelot mathematically links Admin Relief to Patient Safety—the connection that produces the simulation’s most significant findings.

The Code

The full A2A + FHIR implementation is under 200 lines of JavaScript. It runs in a browser. No server, no database, no middleware. Everything is in-memory, in a single HTML file.

This isn’t production-ready. But it took an afternoon to build, and it produces a functioning multi-agent healthcare mesh that can run 200 simulations with full statistical analysis. If the protocol were standardized across real Dutch providers, the integration cost for the IZA transformation would look very different.

The Technical Bet
We aren’t testing if AI can “think.” We are testing if Interoperable Action—agents that can discover each other and exchange structured data without friction—can offset a 301,000-worker shortage.
Wegiz-Ready by Design
Cammelot is aligned with the Wegiz (Wet elektronische gegevensuitwisseling in de zorg), which mandates electronic data exchange for all Dutch healthcare providers. Every agent interaction uses HL7 FHIR R4 resources aligned with Nictiz nl-core profiles.
Technical details: FHIR_STORE is a capped array (2,000 resources) with FIFO eviction. A2A_MESSAGES caps at 200. Agent cards are static in current implementation; dynamic publishing with real-time wait time updates is planned. FHIR resources follow R4 structure but don’t validate against full FHIR schemas—a known shortcut.
